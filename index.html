Automation Scenarios

5가지 자동화 시나리오

SCENARIO 01
Auto Code Review

PR이 생성되면 자동으로 코드 리뷰를 수행합니다.

  • 코드 품질 분석
  • 보안 취약점 탐지
  • 성능 개선 제안
SCENARIO 02
Auto Bug Fix

bug 라벨이 붙은 이슈를 자동으로 분석합니다.

  • 버그 원인 분석
  • 수정 코드 제안
  • 테스트 케이스 추천
SCENARIO 03
Auto Documentation

새 파일이 추가되면 자동으로 문서를 생성합니다.

  • JSDoc 자동 생성
  • 사용 예제 추가
  • README 업데이트
SCENARIO 04
Test Coverage

테스트가 필요한 코드에 테스트를 제안합니다.

  • 테스트 케이스 생성
  • 엣지 케이스 식별
  • 모킹 전략 제안
SCENARIO 05
Dependency Review

의존성 업데이트 PR을 자동으로 검토합니다.

  • 보안 취약점 검사
  • Breaking Changes 분석
  • 호환성 확인

Implementation Guide

핵심 기술 요소

GITHUB_TOKEN Limitation Solution

flowchart LR subgraph Problem[Problem: Cannot Trigger] P1[Event] --> P2[Workflow-1] P2 --> P3[Comment] P3 -.->|X| P4[Workflow-2] end subgraph Solution[Solution: workflow_call] S1[Event] --> S2[Wrapper] S2 --> S3[claude.yml] S3 --> S4[ai-actions] S4 --> S5[Result] end style P4 fill:#ef4444,color:#fff style S3 fill:#10b981,color:#fff style S4 fill:#10b981,color:#fff
GITHUB_TOKEN Limitation: GITHUB_TOKEN은 보안상 새로운 워크플로우를 트리거할 수 없습니다. workflow_call 패턴으로 직접 호출하여 해결합니다.

Required Configuration

# Repository Secrets
CLAUDE_CODE_OAUTH_TOKEN: "your-token"

# Workflow call with secrets
jobs:
  call-claude:
    uses: ./.github/workflows/claude.yml
    with:
      prompt: "Review this PR"
      target_number: ${{ github.event.pull_request.number }}
    secrets: inherit

Step-by-Step Guide

실제 프로젝트에 ai-actions를 적용하기 위한 단계별 진행 가이드

1. 환경 구성

1

Claude 인증 설정 (3가지 방법 중 택 1)

아래 3가지 인증 방법 중 프로젝트 상황에 맞는 것을 선택합니다.

A. /install-github-app (권장)

Claude Code CLI에서 /install-github-app 명령을 실행하면 GitHub App 설치와 Secret 등록이 자동으로 진행됩니다. 가장 간편한 공식 권장 방법입니다. 

# Claude Code 터미널에서 실행
$ claude
> /install-github-app

# 안내에 따라 GitHub App 설치 + Secret 자동 등록
# Repository admin 권한 필요

# claude.yml에서 사용
claude_code_oauth_token: ${{ secrets.CLAUDE_CODE_OAUTH_TOKEN }}
  • Pro/Max 구독 비용만으로 사용 가능
  • 별도 API 과금 없음
  • GitHub App + Secret 자동 구성
GitHub Enterprise: 사내 GHES 환경에서는 이 명령이 동작하지 않을 수 있습니다. 방법 C를 사용하세요.

B. .credentials 파일 활용 (사내 환경 권장)

로컬에서 Claude Code에 한번이라도 로그인하면 ~/.claude/.credentials.json 파일에 OAuth 토큰이 자동 저장됩니다. 사내 GHES 환경에서는 이 방법을 권장합니다. 

# 로컬에서 credentials 확인
$ cat ~/.claude/.credentials.json
# {
#   "claudeAiOauth": {
#     "accessToken": "...",
#     "refreshToken": "...",
#     "expiresAt": "..."
#   }
# }

# accessToken 값을 GitHub Secret에 등록
Name:  CLAUDE_CODE_OAUTH_TOKEN
Value: (accessToken 값 그대로 복사)

# macOS의 경우 Keychain에서도 확인 가능
# Keychain Access > claude 검색 > 암호 보기
  • 별도 발급 절차 없이 기존 토큰 재활용
  • 로컬 Claude Code 로그인만 되면 OK
방법 B (.credentials 파일) 활용 시 제약사항:
  • 토큰 만료: .credentials의 OAuth 토큰은 약 24시간 후 만료됩니다. 만료 시 로컬에서 claude 명령을 다시 실행하여 토큰을 갱신하고 Secret을 업데이트해야 합니다.
  • 자동 갱신 불가: GitHub Actions 환경에서는 토큰 자동 갱신(refresh)이 되지 않으므로, 장기 운영에는 부적합합니다.
  • 보안 주의: .credentials 파일에는 민감한 인증 정보가 포함되어 있으므로, 절대 Git에 커밋하지 마세요. .gitignore에 반드시 포함시켜야 합니다.
  • 개인 토큰: 해당 토큰은 개인 계정에 연결되어 있어, 팀 공유 시 특정 개인의 사용량으로 집계됩니다.
  • 용도 제한: OAuth 토큰은 Claude Code 전용으로, 일반 Anthropic API 호출에는 사용할 수 없습니다.

2

GitHub Repository Secret 등록

Repository > Settings > Secrets and variables > Actions에서 선택한 인증 방법의 키를 등록합니다. Organization 레벨에서 등록하면 모든 레포에서 사용 가능합니다.

# Settings > Secrets > Actions > New repository secret
# 방법 A:   CLAUDE_CODE_OAUTH_TOKEN (자동 등록됨)
# 방법 B:   CLAUDE_CODE_OAUTH_TOKEN (.credentials에서 추출)
3

Core Workflow (claude.yml) 배치

.github/workflows/claude.yml 파일을 생성합니다. 이 파일은 모든 시나리오에서 재사용되는 핵심 워크플로우입니다. workflow_call 트리거, issue_comment 트리거, pull_request_review_comment 트리거를 포함합니다.

4

CLAUDE.md 설정 (선택)

프로젝트 루트에 CLAUDE.md를 생성하여 Claude에게 프로젝트 컨텍스트, 코딩 컨벤션, 리뷰 기준 등을 지시할 수 있습니다. 프로젝트별 맞춤 규칙을 정의하면 리뷰 품질이 향상됩니다.

2. 시나리오별 적용 방법

Scenario 1 적용: Auto Code Review

.github/workflows/workflow-example-1-auto-review.yml 파일을 생성하고, PR 생성 시 자동으로 claude.yml을 호출하도록 설정합니다.

name: Auto Code Review
on:
  pull_request:
    types: [opened, synchronize]
    paths: ['**/*.js', '**/*.ts']

jobs:
  review:
    uses: ./.github/workflows/claude.yml
    with:
      prompt: |
        이 PR의 코드를 리뷰해주세요.
        코드 품질, 보안, 성능을 검토하고
        gh pr comment로 결과를 게시하세요.
      target_number: ${{ github.event.pull_request.number }}
      target_type: pr
    secrets: inherit
적용 팁: paths 필터를 프로젝트에 맞게 조정하세요. 프론트엔드 프로젝트라면 **/*.tsx, 백엔드라면 **/*.py 등으로 설정합니다.

Scenario 2 적용: Auto Bug Fix

Issue에 bug 라벨을 추가하면 자동으로 분석이 시작됩니다.

name: Auto Bug Fix
on:
  issues:
    types: [labeled]

jobs:
  analyze:
    if: github.event.label.name == 'bug'
    uses: ./.github/workflows/claude.yml
    with:
      prompt: |
        이 버그 이슈를 분석하고 원인과
        수정 방안을 gh issue comment로 게시하세요.
      target_number: ${{ github.event.issue.number }}
      target_type: issue
    secrets: inherit

Scenario 3 적용: Auto Documentation

main 브랜치에 새 파일이 push되면 문서화 이슈를 자동 생성합니다.

name: Auto Documentation
on:
  push:
    branches: [main]
    paths: ['src/**/*.js']

jobs:
  create-doc-issue:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Create doc issue
        run: |
          gh issue create \
            --title "Docs: 새 파일 문서화 요청" \
            --body "@claude 새로 추가된 파일을 분석하고 문서화해주세요."
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
주의: ai-actions는 push 이벤트를 직접 지원하지 않으므로 Issue를 생성하여 @claude 멘션으로 우회합니다.

Scenario 4 적용: Test Coverage

테스트 없는 코드가 포함된 PR에 자동으로 테스트 코드를 제안합니다.

name: Test Coverage
on:
  pull_request:
    types: [opened, labeled]

jobs:
  suggest-tests:
    if: |
      contains(github.event.pull_request.title, 'test') ||
      github.event.label.name == 'needs-tests'
    uses: ./.github/workflows/claude.yml
    with:
      prompt: |
        이 PR의 코드에 대한 테스트 코드를 제안하세요.
        엣지 케이스와 경계값 테스트도 포함하세요.
      target_number: ${{ github.event.pull_request.number }}
      target_type: pr
    secrets: inherit

Scenario 5 적용: Dependency Review

의존성 변경이 포함된 PR을 자동으로 보안 검토합니다.

name: Dependency Review
on:
  pull_request:
    types: [opened, synchronize]
    paths: ['package.json', 'package-lock.json']

jobs:
  review-deps:
    if: |
      contains(github.event.pull_request.title, 'bump') ||
      contains(github.event.pull_request.title, 'deps')
    uses: ./.github/workflows/claude.yml
    with:
      prompt: |
        이 PR의 의존성 변경사항을 검토하세요.
        보안 취약점, Breaking Changes, 호환성을 확인하세요.
      target_number: ${{ github.event.pull_request.number }}
      target_type: pr
    secrets: inherit

테스트 PR/Issue 생성

Scenario 1: Auto Code Review - 업무 효율화 사례

기존 방식: 10명 규모 개발팀에서 하루 평균 5건의 PR이 생성됩니다. 시니어 개발자 2명이 번갈아 리뷰하며, PR당 평균 45분이 소요됩니다. 시니어 개발자의 하루 리뷰 시간은 약 2시간 이상이며, 리뷰 대기로 인해 PR 머지까지 평균 6시간이 걸립니다.

자동화 후: Claude가 PR 생성 즉시 1차 리뷰를 수행합니다. 보안 취약점, 코딩 컨벤션 위반, 잠재 버그를 2~3분 내에 코멘트로 게시합니다. 시니어 개발자는 AI 리뷰 결과를 참고하여 10~15분 내에 최종 승인만 하면 됩니다.
